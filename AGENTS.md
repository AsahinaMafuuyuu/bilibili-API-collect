# API Docs Rebuild Agents Guide

本文件用于约束 Agent 对接口文档进行企业级规范化重构。
目标是：可执行、可追踪、可检查、可提交、可持续迭代。

## 1) 范围与总原则

1. 重构范围只针对 `docs/**` 的接口文档。
2. 原始 `docs/**` 不直接覆盖，作为参考源保留。
3. 所有重构产物必须写入 `Rebuild-Api-Docs/**`。
4. 目录映射必须一致：`docs/a/b.md -> Rebuild-Api-Docs/a/b.md`。
5. 每次执行都必须先读 `TODO.md`，再开始处理。
6. 每次执行结束都必须回写 `TODO.md`。
7. 批次必须可独立提交与回滚。

## 2) 文件数量与 Token 硬规则

1. 每次重构的接口文件数量上限为 10。
2. 任何情况下都不得超过 10 个接口文件。
3. 每批次文件数量由“当前会话剩余 token”决定。
4. 推荐映射策略：
5. `>=120k`：最多 10 个文件。
6. `80k-119k`：最多 8 个文件。
7. `50k-79k`：最多 6 个文件。
8. `30k-49k`：最多 4 个文件。
9. `<30k`：最多 2 个文件。
10. 若无法准确获取 token 余额，默认处理 3 个文件。
11. 除接口文件外，允许同步修改 `TODO.md` 和测试脚本。

## 3) 重构输出规则

1. 重构后文档必须符合企业级 API 文档结构。
2. 每个接口需明确：路径、方法、鉴权、参数、返回、错误码。
3. 必须提供请求示例、成功响应示例、失败响应示例。
4. 字段定义需明确类型、必填性、约束、默认值、枚举说明。
5. 统一状态码与错误对象格式，避免同义字段混用。
6. 需要支持传统开发 API 风格，适配团队协作场景。
7. 文档应可导入 Postman、Apifox 等工具。
8. 机器可读优先采用 OpenAPI 3.0/3.1（YAML/JSON）。
9. 如涉及 gRPC，要补充 proto 对应关系与映射说明。

## 4) 批次执行流程

1. 读取 `TODO.md`，筛选仍为 `[ ]` 的条目。
2. 根据第 2 节规则计算本批次目标数量 `N`。
3. 确保 `1 <= N <= 10`。
4. 建立源文件到目标文件映射表。
5. 在 `Rebuild-Api-Docs/**` 生成对应重构文件。
6. 执行格式检查与结构检查。
7. 修复检查问题后再次运行检查。
8. 检查通过后更新 `TODO.md` 勾选状态。
9. 记录本批次摘要（范围、数量、风险、备注）。
10. 本地 Git 提交并写规范提交信息。

## 5) 检查与测试规则

1. 每次批次结束必须执行检查，不可跳过。
2. 可维护校验脚本：`tests/validate-rebuild-docs.ps1`。
3. 至少检查以下内容：
4. 路径镜像是否正确（源/目标目录对应）。
5. OpenAPI 是否语法可解析、核心字段完整。
6. Markdown 结构是否完整（标题、表格、代码块）。
7. 示例是否可读且字段命名一致。
8. `TODO.md` 的勾选是否与本批次一致。
9. 检查不通过时，禁止提交。

## 6) Git 分支与提交规范

1. 分支命名建议：`docs/rebuild-batch-<date>-<index>`。
2. 提交信息遵循 Conventional Commits。
3. 推荐提交格式：
4. `docs(api): rebuild <N> docs into Rebuild-Api-Docs (batch <k>)`
5. 一个批次至少一个清晰提交。
6. 禁止混入与本批次无关的改动。
7. 提交前确认测试通过且 TODO 已更新。

## 7) 连续迭代规则（新进程必遵守）

1. 新进程开始时，先读取 `TODO.md`。
2. 只从未完成条目中选取任务。
3. 继续遵守 token 决策与 `<=10` 上限。
4. 不重复处理已经 `[x]` 的文件。
5. 若出现规则冲突，以本文件硬规则优先。
6. 若遇不确定项，采取保守策略（减少文件数）。

## 8) 交付定义（DoD）

1. 重构文件已落在 `Rebuild-Api-Docs/**` 且结构正确。
2. 格式与结构检查通过。
3. `TODO.md` 已更新并可作为下次起点。
4. 本地 Git 已完成规范化提交。
5. 批次可复盘、可追踪、可继续。

## 9) 执行报告输出（必做）

1. 每次执行完当前进程任务后，必须输出一次任务报告。
2. 报告目录固定为项目根目录 `Aengt-Report/`（若不存在则创建）。
3. 报告文件命名：`<企业级别规范时间>.md`（建议使用 `YYYY-MM-DD_HHmm.md`）。
4. 目录结构要求：
5. `Aengt-Report/<企业级别规范时间>.md`
6. 报告内容保持精简，总行数控制在 40 行以内。
7. 报告至少包含：本次处理内容、完成度、测试通过情况、其他备注（如有）。
8. 若存在不确定点，必须写入 `Question` 小节，且问题需与本项目相关。
9. `Question` 可为空；若本次进展顺利可不写具体问题。
10. Question 模板如下（`A1` 留空，等待人工补充）：
11.
12. ```markdown
13. ## Question
14. - Q1: 对于live中https://www.bilibili.api/v1/user/info api进行测试的时候发现api参数有问题，请告诉我下一步该怎么处理这个问题
15. - A1:
16. ```
